# MCP Go SDK Selection (2026-05-26)

**Chosen:** `github.com/mark3labs/mcp-go` v0.54.1

**Considered:**
- `github.com/modelcontextprotocol/go-sdk` v1.6.1 (official)
- `github.com/mark3labs/mcp-go` v0.54.1 (community)

**Why:**
- **Client API maps directly to our `Transport` interface** — `client.Client` exposes `Initialize`, `ListTools`, `CallTool`, `Ping`, `Close` with signatures that align one-to-one with our abstraction. The official SDK uses a session-based model (`client.Connect` returns a `session`) which forces us to hold two objects and map session methods to our interface.
- **Server side is trivial** — `server.NewMCPServer` + `server.ServeStdio` lets the echo test fixture ship in <60 lines. The official SDK requires schema-annotated structs and generic `AddTool` handlers, adding boilerplate for a throwaway fixture.
- **Subprocess lifecycle is fully controllable** — `transport.NewIO` accepts existing `io.Reader`/`io.WriteCloser` streams, so we spawn `exec.Cmd` ourselves, capture PID for `ProcessInfo()`, and implement our own SIGTERM→3s grace→SIGKILL shutdown sequence. The official SDK's `CommandTransport` encapsulates the child and does not expose PID or signal control.
- **HTTP/SSE transports already exist** — `client.NewSSEMCPClient` and `client.NewStreamableHttpClient` are ready for PR-C; no second SDK will be needed.
- **Zero transitive dependency footprint** — `go get` adds only the single module (2 go.sum lines). The official SDK is similarly light, so this is not a differentiator, but mcp-go wins on API fit.

**Tradeoffs accepted:**
- Version is 0.x (v0.54.1). API has been stable across the last 20+ minor releases, but we pin to an exact version and will revisit at v1.0.
- `transport.Stdio` fields are unexported, so we use `transport.NewIO` + our own `exec.Cmd` to obtain PID rather than relying on SDK-exposed process handles.

## Update 2026-05-26 (PR-C verification)

- **HTTP client in SDK:** ✅ `client.NewStreamableHttpClient(baseURL, ...transport.StreamableHTTPCOption)` available in v0.54.1. Internally creates `transport.NewStreamableHTTP` + `client.NewClient`. Supports headers via `WithHTTPHeaders`, custom `http.Client` via `WithHTTPBasicClient`, session tracking via `Mcp-Session-Id`.
- **SSE client in SDK:** ✅ `client.NewSSEMCPClient(baseURL, ...transport.ClientOption)` available in v0.54.1. Internally creates `transport.NewSSE` + `client.NewClient`. Supports headers via `WithHeaders`, custom `http.Client` via `WithHTTPClient`, endpoint discovery via SSE `event: endpoint`.
- **PR-C path:** wrap SDK for both transports. Each transport holds `*client.Client`; `Connect` calls `client.Start(ctx)` then `client.Initialize(ctx, req)` just like stdio. `Close` calls `client.Close()`. No hand-rolling needed.
- **Rationale for wrapping vs hand-rolling:** SDK implements full MCP protocol over HTTP/SSE including JSON-RPC framing, session IDs, SSE event parsing, endpoint resolution, OAuth, and error handling. Re-implementing would duplicate ~1,500 LOC with no benefit and higher drift risk.

**Fallback plan:**
- If mcp-go breaks stdio client API in a future minor version or stops receiving updates, switch to the official SDK (`github.com/modelcontextprotocol/go-sdk`). The impact is contained to `transport_stdio.go` and the echo fixture; `hypervisor.go` and `handlers/mcp.go` do not import the SDK directly.
